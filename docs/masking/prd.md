# 数据脱敏（Masking）产品需求文档

## 1. 背景

在数据处理和展示过程中，敏感信息（手机号、身份证、姓名、银行卡、邮箱、地址等）需要脱敏处理以防止隐私泄露。脱敏模块通过字段名自动推断敏感类型，并应用格式保留的掩码规则。

## 2. 目标

- 基于字段名关键字自动识别敏感类型。
- 对常见 PII 提供默认掩码规则，保持数据格式。
- 支持单字段、整记录、批量字段多种调用方式。
- 提供 HMAC 哈希与字符串截断作为补充工具。

## 3. 功能需求

### 3.1 核心功能

| 编号 | 需求 | 优先级 |
|---|---|---|
| MSK-01 | 字段名自动推断敏感类型 | P0 |
| MSK-02 | 手机号脱敏（保留前3后4） | P0 |
| MSK-03 | 身份证脱敏（保留前6后4） | P0 |
| MSK-04 | 姓名脱敏（保留首字） | P0 |
| MSK-05 | 银行卡脱敏（保留前4后4） | P0 |
| MSK-06 | 邮箱脱敏（保留首尾+域名） | P0 |
| MSK-07 | 地址脱敏（保留前6字符） | P0 |
| MSK-08 | 整记录脱敏 | P0 |
| MSK-09 | 批量字段脱敏 | P1 |
| MSK-10 | HMAC-SHA256 哈希 | P1 |
| MSK-11 | 字符串截断 | P1 |

### 3.2 非功能需求

| 编号 | 需求 | 指标 |
|---|---|---|
| MSK-NF01 | 低延迟 | 单次脱敏 < 0.1ms |
| MSK-NF02 | 无外部依赖 | 纯本地计算 |
| MSK-NF03 | 线程安全 | 并发调用无竞争 |

## 4. 用户故事

### 4.1 API 响应脱敏

> 作为后端开发者，我希望在 REST 接口返回前对用户敏感字段自动脱敏，确保前端展示安全。

### 4.2 日志脱敏

> 作为运维工程师，我希望在写入日志前对请求参数中的敏感字段脱敏，防止日志泄露用户隐私。

## 5. 验收标准

- [ ] 手机号脱敏：`13812345678` → `138****5678`
- [ ] 身份证脱敏：`110101199001011234` → `110101********1234`
- [ ] 姓名脱敏：`张三` → `张*`，`张三丰` → `张**丰`
- [ ] 银行卡脱敏：`6222021234567890123` → `6222 **** **** 0123`
- [ ] 邮箱脱敏：`zhangsan@example.com` → `z***n@example.com`
- [ ] 地址脱敏：`北京市朝阳区某某街道123号` → `北京市朝阳区****`
- [ ] 整记录脱敏不修改原记录
- [ ] 批量脱敏长度校验

## 6. 约束与限制

- 字段类型推断基于关键字匹配，非 NLP 语义理解
- 不支持自定义脱敏规则（需扩展代码）
- 短于保留位数的值直接返回原值
